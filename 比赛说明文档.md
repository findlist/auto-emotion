## 二、项目启动流程

### 2.1 五步开发法

```
Step 1: 创意构思 → 确定赛道、痛点、产品形态
Step 2: 需求设计 → 编写 project-spec.md 项目规格说明
Step 3: 任务拆解 → 编写 tasks.md 任务清单 + checklist.md 检查清单
Step 4: 开始开发 → 按任务清单并行/串行开发
Step 5: 审查验收 → 功能审查 + 代码审查 + 修复优化
```

### 2.2 Step 1：创意构思

回答以下 4 个核心问题：

1. **想解决什么问题**：用 1-2 句话描述痛点或需求
2. **为什么会想到做这个**：简单说明动机或灵感来源
3. **大概是什么产品**：一句话概括产品形态（App/小程序/网站/系统/硬件）
4. **价值与意义**：从商业价值、效率提升、社会价值中选 1-2 个角度

### 2.3 Step 2：需求设计

编写 `docs/project-spec.md`，包含：

- 项目背景（社会背景 + 赛事背景 + 项目定位）
- 系统架构（架构图 + 技术栈选型）
- 统一规范（API 响应格式、错误码、命名规范、目录结构）
- 业务模块设计（各模块功能清单 + 数据模型）

### 2.4 Step 3：任务拆解

使用 Spec 模式生成 3 个文件：

| 文件 | 作用 |
|------|------|
| `spec.md` | 功能规格说明（Why + What Changes + Requirements） |
| `tasks.md` | 任务清单（有序、可验证、标注依赖关系） |
| `checklist.md` | 检查清单（逐项验收标准） |

**任务拆解原则**：
- 每个任务是一个小而可验证的工作单元
- 标注任务间依赖关系（哪些可并行、哪些必须串行）
- 基础模块（用户体系）优先开发，业务模块可并行

### 2.5 Step 4：开始开发

**依赖分析示例**：

```
第一阶段（串行）：用户体系（注册/登录/鉴权/个人资料）
    ↓
第二阶段（可并行）：业务模块 A | 业务模块 B | 业务模块 C
    ↓
第三阶段（串行）：集成测试 + 审查优化
```

### 2.6 Step 5：审查验收

审查 4 大维度：

1. **功能完整性**：Checklist 逐项核查
2. **流程闭环性**：从用户操作到数据落地的全链路验证
3. **技术架构合理性**：分层规范、公共能力复用
4. **风险可控性**：数据一致性、幂等性、权限控制、异常兜底

## 四、统一规范

### 4.1 API 响应格式

```json
// 成功
{ "code": 200, "message": "操作成功", "data": {} }

// 分页
{ "code": 200, "message": "查询成功", "data": {
    "list": [], "total": 100, "page": 1, "pageSize": 20,
    "totalPages": 5, "hasNext": true
}}

// 错误
{ "code": 400, "message": "参数错误", "errors": [
    { "field": "phone", "message": "手机号格式不正确" }
]}
```

### 4.2 错误码规范

| 错误码 | 说明 |
|--------|------|
| 200 | 成功 |
| 201 | 创建成功 |
| 400 | 请求参数错误 |
| 401 | 未授权 |
| 403 | 权限不足 |
| 404 | 资源不存在 |
| 422 | 参数验证失败 |
| 429 | 请求过于频繁 |
| 500 | 服务器内部错误 |

### 4.3 数据库迁移规范

- 文件命名：`001_init.sql`、`002_xxx.sql`、`003_xxx.sql`
- 所有 DDL 语句必须幂等：`CREATE TABLE IF NOT EXISTS`、`ADD COLUMN IF NOT EXISTS`、`DROP CONSTRAINT IF EXISTS`、`CREATE INDEX IF NOT EXISTS`
- 每个迁移文件聚焦一个功能域，不混合多个不相关的变更

### 4.4 代码规范

- 前后端均使用 TypeScript，启用严格模式
- 后端分层：路由层 → 服务层 → 数据层，职责分离
- 所有 SQL 使用参数化查询，禁止字符串拼接
- 密码使用 bcrypt 加盐哈希
- 核心资产变更接口必须使用数据库事务 + 幂等控制

## 五、P0 级质量要求（上线前必须完成）

### 5.1 数据安全

- [ ] 所有涉及资产变更（积分、时长、金额）的接口使用数据库事务
- [ ] 事务内查询使用 `FOR UPDATE` 行锁，防止并发写脏
- [ ] 核心接口（下单、结算、退款）实现幂等控制（5 秒窗口去重）

### 5.2 权限控制

- [ ] 全局认证中间件覆盖所有业务接口
- [ ] 水平权限校验：用户只能操作自己的订单和数据
- [ ] 角色权限分级：普通用户 / 管理员，管理后台独立路由
- [ ] JWT 登出加入黑名单，封禁用户立即失效

### 5.3 异常兜底

- [ ] 订单超时自动取消/完成（定时任务）
- [ ] 拼单超时未成团自动退款
- [ ] 全局异常捕获，不暴露后端堆栈信息
- [ ] WebSocket 断线自动重连 + 离线消息补齐

### 5.4 安全加固

- [ ] 密码 bcrypt 加密存储
- [ ] 手机号格式正则校验
- [ ] 敏感接口限流（登录、注册、下单）
- [ ] 环境变量启动校验（JWT_SECRET 等必填项）

## 七、部署方案

### 7.1 Docker 一键部署

```bash
# 1. 配置环境变量
cp .env.example .env
# 编辑 .env，填写 JWT_SECRET 和 DB_PASSWORD

# 2. 一键启动
docker compose up -d

# 3. 访问
# 前端：http://localhost
# API：http://localhost:3000/api
# 健康检查：http://localhost:3000/health
```

### 7.2 本地开发

```bash
# 终端 1：启动后端（热重载）
cd server && npm install && npm run dev

# 终端 2：启动前端（Vite）
cd client && npm install && npm run dev

# 数据库迁移
psql -h localhost -U postgres -d your_db -f database/migrations/001_init.sql
```
