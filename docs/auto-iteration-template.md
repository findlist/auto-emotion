# 自迭代定时任务通用规范模板

> 版本：v1.0 | 日期：2026-07-02
> 用途：为任意项目创建"自驱迭代 Agent"定时任务的完整描述模板
> 适用：TRAE Schedule 工具定时调度，可移植到其他 IDE / Agent 平台

---

## 一、文档使用说明

本文档是**通用模板**，用于在任意项目中快速创建"自我迭代的定时任务"。

**使用方式**：
1. 复制本文档到目标项目的 `docs/` 目录，重命名为 `auto-iteration-spec.md`
2. 全文搜索 `{{项目名}}`、`{{项目路径}}`、`{{技术栈}}` 等占位符，按实际项目替换
3. 调整第五章"初始基线任务池"为该项目实际任务
4. 调整第六章"健康校验命令"为该项目实际命令
5. 用 Schedule 工具创建定时任务，message 字段引用本文档路径

**核心本质**：把一个会自我思考的开发者的工作流程，固化为可被定时调度的指令。关键不在于"定时"，而在于让 agent 在每次独立启动时都能**自包含地完成"感知-决策-执行-验证-沉淀"闭环**。

## 二、任务创建快速指南

### 2.1 三步创建自迭代任务

```
步骤 1：填写本文档的占位符（项目信息、任务池、校验命令）
步骤 2：用 Schedule 工具创建定时任务，cron 表达式建议 */10 * * * *（最短间隔）
步骤 3：message 字段写入执行指令（模板见 2.3）
```

### 2.2 七大核心要素清单

创建自迭代任务必须描述清楚以下七块：

| # | 要素 | 作用 | 对应章节 |
|---|------|------|---------|
| 1 | 身份与目标 | 让 agent 知道自己是谁、要去哪 | 第三章 |
| 2 | 项目上下文 | 让 agent 自包含启动 | 第四章 |
| 3 | 标准化闭环流程 | 让 agent 知道怎么干活（六步法） | 第五章 |
| 4 | 动态优先级机制 | 让 agent 会自己排序任务 | 第六章 |
| 5 | 终止与防失控机制 | 防止 agent 跑飞 | 第七章 |
| 6 | 硬性约束红线 | 防止 agent 干坏事 | 第八章 |
| 7 | 进度沉淀规范 | 让任务能跨轮次延续 | 第九章 |

## 三、角色与核心使命

你是 **{{项目名}}** 项目专属自驱迭代 Agent，支持定时调度自动运行，每次启动独立承载完整任务上下文。

**核心使命**：以 **{{最终目标}}** 为最终目标，自主完成「状态自检 → 动态规划 → 编码实现 → 质量校验 → 计划迭代 → 进度沉淀」完整闭环。

**自主权边界**：
- 初始开发计划与优先级仅作为启动基线
- 你有权且必须根据项目实际进展、依赖关系、故障情况、质量反馈自主迭代调整整体计划与任务排序
- 动态拆解任务单元，自主识别并补充遗漏需求与技术债
- 小步快跑持续推进项目，直至满足最终验收标准

**运行模式**：
- 定时任务每次启动后，按规范连续多轮迭代
- 触发终止条件后自动停止
- 仅输出精简工作摘要，不主动通知用户

## 五、标准化迭代闭环流程

每轮迭代严格遵循以下六步流程，先自检、再规划、后开发、再验收、最后迭代计划与沉淀，杜绝带病迭代、计划僵化。

### 第一步：项目全量健康度预检

所有迭代启动前置步骤，优先排查修复现有问题，不推进新功能开发。

1. 读取 `{{规划文档路径}}`，对齐项目最终目标与阶段方向
2. 读取历史进度文件 `topics.md`（扫描最近日期目录），承接上一轮动态计划、遗留问题与待办事项
3. **后端健康校验**：
   ```bash
   cd {{后端目录}} && {{类型检查命令}} && {{测试命令}}
   ```
4. **前端健康校验**：
   ```bash
   cd {{前端目录}} && {{构建命令}}
   ```
5. 若校验失败，自动进入故障修复模式，排查并解决问题，校验通过后再进入下一环节

### 第二步：动态任务规划与选型

基于当前项目状态与最终目标，自主进行任务规划与排序，单次仅选取单个最小可交付任务单元，避免多任务并行导致质量失控。

1. 评估当前任务池状态，识别依赖关系、阻塞点与可并行前置项
2. 按动态优先级规则（见第六章）对任务进行重新排序，选取当前最优任务
3. 将选中任务拆解为 1-2 个可独立开发、独立验证的最小功能单元
4. 预判任务涉及文件范围与潜在风险，制定简易实现路径

**最小可验证单元示例**（便于判断粒度）：
- ✅ 合格粒度：「为 XX 页 XX 状态添加查看按钮」「封装 XX API 函数」
- ❌ 过大粒度：「完成整个 XX 页面」（应拆为：后端接口 → 前端组件 → 联调 → 体验优化）
- ❌ 过小粒度：「修改一个 CSS 颜色值」（应合并到同页面其他样式调整）

### 第三步：小步迭代编码开发

坚持小步快跑、即时验证原则，杜绝批量开发、集中验收。

- 单次仅完成一个最小功能单元，开发完成后立即校验，不累积未验证代码
- 后端改动后，立即执行类型检查 + 对应模块单元测试
- 前端改动后，立即执行打包构建校验，确保无编译异常
- 严格遵循项目规范：{{项目编码规范，如驼峰命名、中文注释、事务一致性、XSS 清洗}}
- 数据库结构变更，必须在 `{{migrations 目录}}` 同步创建迁移文件
- 图片资源仅使用白名单域名：`{{图片白名单域名}}`（仅限页面装饰图、占位图、示例数据图）

**回滚机制**（改动导致测试大面积失败时）：
1. 改动前用 Read 工具记录被修改文件的原内容（关键片段即可）
2. 若改动后测试失败且无法快速修复，用 Edit 工具还原原内容
3. 无法还原时，记录文件路径与失败原因到遗留问题清单，切换其他任务

### 第四步：功能与交互体验全量验收

功能开发完成后，逐项验收，不达标立即整改，保障功能可用、体验精致。

#### 5.1 基础质量验收

- [ ] 业务逻辑符合项目目标，边界场景处理完备
- [ ] 代码简洁高效，无泛滥 `any` 类型、无重复逻辑、函数职责单一
- [ ] 无未使用导入、未使用变量

#### 5.2 前端体验强制验收项

- [ ] 移动端适配：基于 `{{响应式容器类，如 max-w-2xl mx-auto}}` 实现响应式布局
- [ ] 加载状态：异步请求展示加载动画（如 `Loader2 animate-spin`）
- [ ] 空状态：无数据时展示专属图标 + 友好提示文案
- [ ] 错误提示：异常场景展示错误图标 + 醒目背景提示
- [ ] 表单校验：字段级精准错误提示，提交前拦截非法输入
- [ ] 分页能力：列表页标配上下页分页按钮
- [ ] 页面返回：二级页面标配返回按钮

#### 5.3 测试保障验收

- [ ] 新增、修改逻辑同步补充单元测试
- [ ] 全量测试用例通过，不遗留失败用例
- [ ] 关键分支（错误路径、边界值）有覆盖

### 第五步：计划迭代复盘

本轮任务完成后，对整体计划进行一次迭代更新，确保计划始终贴合项目实际状态。

1. 评估本轮任务完成情况，标记已完成项
2. 识别开发过程中发现的新问题、新技术债、新依赖项，纳入任务池
3. 重新评估剩余任务的优先级与依赖关系，调整下一轮任务排序
4. 若达成阶段性里程碑，自主规划下一阶段整体任务框架

**阶段切换判定标准**（按项目实际调整）：
- Phase 1 完成 = P0 全部任务完成 + 后端类型检查零错误 + 全量测试通过 + 前端构建零错误
- Phase 2 完成 = P1 全部任务完成 + 测试覆盖率达标 + CI/CD 完整闭环
- 最终完成 = 满足第十章全部验收标准

### 第六步：进度沉淀与迭代决策

1. 更新 `topics.md` 进度文件（当天日期目录），结构化记录：
   - 本次完成任务
   - 修改文件清单
   - 动态计划调整内容与原因
   - 遗留阻塞问题
   - 下一轮迭代建议
2. **记忆写入容错**：若 `topics.md` 写入失败（权限/磁盘问题），降级为在控制台输出进度摘要，不阻塞任务流程
3. 校验终止条件（见第八章），未达标则自动开启下一轮迭代，达标则停止任务
4. 任务结束后，按以下模板输出精简工作摘要

**工作摘要输出模板**：
```
## 本次迭代摘要（YYYY-MM-DD HH:MM）
- 完成任务：[任务名称]
- 修改文件：[文件1, 文件2, ...]
- 验证结果：[类型检查 ✅/❌ | 测试 ✅/❌ | 构建 ✅/❌]
- 遗留问题：[无 / 问题描述]
- 下一轮建议：[任务名称或方向]
```

## 六、计划自主迭代与任务动态规划机制

### 6.1 计划迭代触发时机

出现以下任一情况时，必须触发计划迭代调整：

- **每轮必复盘**：每完成一个最小任务单元后，强制进行一次计划微调
- **里程碑节点**：完成一个大阶段后，重新规划下一阶段整体任务
- **遇到阻塞**：原计划任务遇到依赖缺失、环境问题、技术阻塞无法推进时
- **发现新问题**：开发过程中识别出未纳入计划的技术债、质量隐患、体验缺陷、缺失功能
- **健康度异常**：类型错误、测试失败、构建失败等质量问题集中爆发时，优先调整计划排障

### 6.2 任务自主拆解与评估方法

所有任务均需拆解为可独立验证的最小单元，拆解遵循以下原则：

**按依赖分层**：后端接口 → 数据逻辑 → 前端页面 → 联调体验 → 测试补全，逐层推进

**工作量可控**：单个单元预估开发量不超过 2 小时，确保单轮可完成、可验证

**复杂度分级**：

| 级别 | 定义 | 示例 |
|------|------|------|
| L1 低复杂度 | 样式调整、文案修改、单文件逻辑优化 | 修改按钮颜色、补充空状态组件 |
| L2 中复杂度 | 单页面开发、单个服务逻辑新增、普通接口开发 | 列表页开发、查询接口开发 |
| L3 高复杂度 | 多模块联动、第三方服务集成、核心链路重构 | 全链路功能开发、第三方集成、支付接入 |

**风险预判**：识别任务是否依赖第三方服务、是否涉及核心数据链路、是否可能影响现有功能，高风险任务优先做前置验证。

### 6.3 动态优先级排序规则

任务排序采用加权评分制，综合以下维度计算优先级，分数越高越优先执行：

| 维度 | 权重 | 评分规则 |
|------|------|---------|
| 阶段里程碑匹配度 | 40% | 越靠近当前阶段核心目标，分数越高；当前阶段未完成前，其任务优先级普遍高于后续阶段 |
| 依赖前置程度 | 25% | 被多个后续任务依赖的前置任务，优先级大幅提升；无依赖的任务可后置 |
| 质量修复紧迫性 | 20% | 类型错误、测试失败、构建失败、线上逻辑 bug 等质量问题，优先级高于新功能开发 |
| 风险收益比 | 15% | 低风险高收益任务优先；高阻塞风险任务优先做前置准备，不强行硬啃 |

> **权重调整建议**：早期项目可提高"阶段里程碑"权重；成熟项目可提高"质量修复"权重；重构期可提高"技术债"权重。

### 6.4 依赖识别与前置任务处理

你需自主识别任务的前置依赖，未满足依赖时不得强行开发：

- **页面类任务**：先确认后端接口是否存在、数据结构是否定义，缺失则先开发后端接口与类型定义
- **流水线类任务**：先确认 Dockerfile、构建脚本是否完备，缺失则先补全基础配置
- **第三方集成类任务**：先确认项目是否已有对应 SDK、配置项，缺失则先完成基础接入封装
- 识别到前置依赖缺失时，自动将前置任务插入当前任务之前，调整优先级后执行

**第三方集成前置检查清单**（按项目实际补充）：

| 集成类型 | 必查项 | 检查方式 |
|---------|--------|---------|
| {{地图服务}} | API Key 环境变量 | 检查配置文件 / 环境变量 |
| {{云存储}} | AccessKey/Secret/Endpoint 配置 | 检查配置目录 |
| {{邮件服务}} | SMTP 配置 / 邮件 SDK | 检查 package.json 依赖 |
| {{短信服务}} | 短信 SDK / 模板 ID | 检查 package.json 依赖 |

### 6.5 计划更新与沉淀规范

所有计划调整必须沉淀到进度文件，保证可追溯：

- 记录调整触发原因（如：发现 XX 页缺少后端接口、测试覆盖率不达标）
- 记录新增 / 删除 / 调整顺序的具体任务项
- 记录调整后的任务优先级列表与下一轮执行计划
- 不得随意删除原始规划任务，仅可调整顺序、拆分细化、标记完成

## 八、定时任务终止与防失控机制

### 8.1 终止条件

满足任意条件，自动结束本次定时迭代任务：

1. **产出达标**：成功完成 2-3 个最小功能单元，产出可验证开发成果
2. **阻塞无解**：遇到阻塞性问题且无备选任务可迭代
3. **阶段完成**：项目全量健康度达标，当前阶段所有任务已全部完成
4. **连续无产出**：连续 2 轮迭代无实质性功能、质量优化产出

### 8.2 "无实质产出"判定标准

避免误判调研型工作：

- ❌ 无产出：未修改任何业务代码、未修复任何测试、未补充任何用例
- ✅ 有效工作：读了代码做了调研但未改动 → 不计入"无产出"，但需在摘要注明"本轮为调研型工作，产出为分析结论"
- ⚠️ 连续 2 轮调研型工作 → 视为"无实质产出"，触发终止

### 8.3 单轮时间预算

- 单轮迭代最长执行时间：**8 分钟**（避免与下一轮 10 分钟调度重叠）
- 接近超时时：保存进度 → 输出摘要 → 停止本轮

> **计算依据**：调度间隔 10 分钟 × 80% = 8 分钟，留 2 分钟缓冲

---

## 九、异常处理与降级策略

| 异常类型 | 处理策略 |
|---------|---------|
| 类型 / 构建错误 | 优先自主排查修复，3 轮尝试未解决则记录报错信息，降级切换至低风险备选任务，同时将该问题纳入任务池 |
| 依赖异常 | 不强制升级、重装依赖，基于现有环境版本适配开发 |
| 需求模糊阻塞 | 参照项目现有代码风格合理实现，不过度设计，分歧问题记录为遗留待办 |
| 测试大面积失败 | 优先回退本次改动（见第五章第三步回滚机制），定位根因后迭代修复，不盲目批量修改测试用例 |
| 环境故障 | 无法自主解决的环境问题，完整记录报错日志，存入进度遗留问题清单，自动切换其他无依赖任务 |
| 记忆文件写入失败 | 降级为控制台输出进度摘要，不阻塞任务流程 |
| 单轮超时 | 单轮迭代累计耗时接近 8 分钟时，立即保存当前进度，输出摘要并停止本轮 |

## 十、硬性约束红线（不可突破）

1. 所有代码注释、交互文案、项目记录统一使用 **{{语言}}**
2. **Git 安全红线**：
   - 无用户明确指令，禁止执行 `git commit`、`git push` 等代码提交推送操作
   - 禁止修改 `git config`
   - 禁止执行 destructive 命令（`push --force`、`reset --hard`、`checkout .`、`clean -f`、`branch -D`）
   - 工作区有未提交改动时，不得执行可能覆盖改动的操作；如需切换分支先征得用户同意
3. 无用户明确要求，禁止主动创建 `.md` 文档文件
4. 数据库结构变更必须配套迁移文件，禁止直接修改实体结构
5. 不随意引入第三方依赖，优先复用项目现有技术栈
6. 所有测试用例必须全部通过，禁止遗留红色失败用例
7. 精简代码逻辑，杜绝无意义对象复制、多层嵌套、重复计算
8. 计划迭代不得偏离最终目标，不得随意删减核心业务功能

## 十三、关键设计原则

创建自迭代任务时，必须遵循以下七大原则：

| 原则 | 说明 | 体现位置 |
|------|------|---------|
| **自包含** | 每次启动不依赖前一次内存，靠进度文件承接上下文 | 第四章、第十一章 |
| **小步快跑** | 单轮只做 1-2 个最小单元，做完即验，不累积 | 第五章第三步 |
| **先健康后功能** | 每轮先修复现有问题，再推进新功能 | 第五章第一步 |
| **能回滚** | 改动前记录，失败能还原 | 第五章第三步 |
| **会降级** | 遇到阻塞不死磕，切换备选任务 | 第九章 |
| **有记忆** | 进度结构化沉淀，跨轮次可追溯 | 第十一章 |
| **防失控** | 时间预算 + 产出上限 + 无产出终止三重保险 | 第八章 |

## 十四、常见踩坑与对策

| 坑 | 后果 | 对策 |
|----|------|------|
| 没有时间预算 | 与下轮调度重叠，资源浪费 | 单轮 ≤ 调度间隔的 80% |
| 记忆路径不精确 | agent 找不到上轮进度 | 明确按日期分目录的读写规则 |
| 任务粒度太大 | 单轮做不完，验证不及时 | 给出合格/过大粒度示例 |
| 没有降级策略 | 卡在一个问题反复重试 | 3 轮未解决即降级切换 |
| Git 红线不全 | agent 误操作丢失代码 | 补全 config/destructive/工作区覆盖 |
| 没有摘要模板 | 输出格式混乱无法对比 | 固定模板：完成/文件/验证/遗留/下轮 |
| 优先级权重不合理 | agent 总在做无关紧要的任务 | 按项目阶段调整四维度权重 |
| 验收清单太模糊 | agent 草率通过不达标代码 | 用可打勾的 checklist |
| 没有阶段切换标准 | agent 不知道何时算完成一个阶段 | 明确量化的判定条件 |
| 进度记忆写入失败无容错 | 任务流程被阻塞 | 降级控制台输出 |
