## 具体修改内容

### 1. 修复已领取状态进度条流光不停止的视觉 Bug（核心）

**问题**：`progress-fill` 工具类一旦应用，其 `::after` 伪元素会持续运行 `progressShimmer` 流光动画（参见 `index.css` 第 305-323 行）。原实现在已领取（归档态）的进度条上仍保留 `progress-fill` 类，导致已领取的任务/成就进度条仍在持续闪烁流光，与"归档/已结束"的语义冲突——已领取状态应当是静态的、不再变化的。

#### 1.1 tasks.tsx · 已领取任务进度条

**修改前**：
```jsx
<div
  className={`h-full rounded-full transition-all progress-fill ${
    status === 'claimed'
      ? 'bg-ink/40'
      : status === 'completed'
      ? 'bg-mint'
      : 'bg-pink'
  }`}
  style={{ width: `${progressPercent}%` }}
/>
```

**修改后**：
```jsx
<div
  // 已领取(归档态)不加 progress-fill 流光，避免静止状态仍有动画干扰；
  // 进行中/可领取才叠加流光暗示进度仍在累积
  className={`h-full rounded-full transition-all ${
    status === 'claimed'
      ? 'bg-ink/40'
      : `progress-fill ${
        status === 'completed'
          ? 'bg-mint'
          : 'bg-pink'
      }`
  }`}
  style={{ width: `${progressPercent}%` }}
/>
```

#### 1.2 achievements.tsx · 已领取成就进度条

同样的修改逻辑：`achievement.claimed` 为 true 时不应用 `progress-fill` 类，仅保留 `bg-ink/40` 静态灰阶背景。

**设计原因**：
- 已领取（claimed）是"归档态"——任务已完成且奖励已领取，进度不应再有动态变化，流光动画在此状态下是误导性的视觉噪音
- 进行中（pending）和可领取（completed）状态保留流光，暗示进度仍在累积或刚到达完成阈值，符合"动态/活跃"语义
- 与上一轮的调色板优化呼应：已领取用 `ink/40` 灰阶（静态/归档），可领取用 `mint`（饱满/活跃），流光动画的有无进一步强化状态语义

### 2. Loading Spinner 风格统一

**问题**：项目内 5 个列表页的 loading spinner 风格不一致：
- `shop.tsx`：`border-4 border-ink/20 border-t-pink`（柔和浅灰环 + pink 顶色）
- `tasks.tsx`、`achievements.tsx`、`leaderboard.tsx`、`season-pass.tsx`：`border-4 border-ink border-t-pink`（强对比深黑环 + pink 顶色）

**修改**：将 4 个页面的 spinner 统一为 `border-4 border-ink/20 border-t-pink`，与 `shop.tsx` 一致。

**设计原因**：
- spinner 是辅助性加载指示器，对比度过强（`border-ink`）在 cream 浅色背景上过于抢眼，与 Neo-brutalism 已建立的"主体硬朗 + 辅助柔和"层次冲突
- `border-ink/20` 浅灰环 + `border-t-pink` 顶色既保留品牌色识别度，又让 spinner 退居辅助位置，不与卡片主内容争夺视觉焦点
- 与 `Loading` 组件（`components/Loading.tsx`）的设计思路一致：组件本身用 `border-cream border-t-pink` 在深色背景上同样保持柔和对比

**未修改**：`shop.tsx` 已是目标风格，无需改动；`Loading` 组件在深色背景上的样式保留原样。

### 3. 首页主入口按钮 emoji 加 hover 放大效果

**问题**：`home.tsx` 主入口双按钮（"挂机空间"🎮 / "对战大厅"⚔️）的 emoji 此前无任何 hover 反馈，与 `shop.tsx` 商品卡片的 emoji hover 放大交互不一致。作为首页最重要的两个入口按钮，缺少微交互反馈显得"沉闷"。

**修改**：按钮加 `group` 类，emoji 加 `transition-transform group-hover:scale-110`：

```jsx
<button
  className="... group"
  ...
>
  <span aria-hidden="true" className="text-4xl transition-transform group-hover:scale-110">🎮</span>
  ...
</button>
```

**设计原因**：
- 与 `shop.tsx` 第 208 行 `transition-transform hover:scale-110` 风格一致，全站 emoji 交互统一
- 用 `group-hover` 而非 `hover` 是因为 emoji 是 button 的子元素，希望"hover 按钮任意位置时 emoji 才放大"，而非"hover emoji 本身"。`group` + `group-hover` 实现准确的父子触发关系
- `transition-transform` 让放大平滑过渡，避免突变
- 与按钮已有的 `hover:translate-x-[2px] hover:translate-y-[2px]`（按下位移）叠加，形成"按钮下沉 + emoji 浮起放大"的双重微交互，提升首页的活力感

### 4. 挂机页升级属性卡片 stagger 入场

**问题**：`idle.tsx` 的"升级角色属性"列表（6 个属性卡片：生命/攻击/防御/暴击率/暴击伤害/挂机效率）外层用 `animate-fadeIn` 整组淡入，但每个卡片缺少 stagger 延迟，6 个卡片同时出现显得突兀，与 `shop.tsx`、`tasks.tsx`、`achievements.tsx`、`leaderboard.tsx`、`season-pass.tsx` 的列表项 stagger 风格不一致。

**修改**：每个属性卡片加 `animate-stagger` 类 + `style={{ animationDelay: `${idx * 50}ms` }}`：

```jsx
{UPGRADE_FIELDS.map((field, idx) => {
  ...
  return (
    <div
      key={field.key}
      // 加 animate-stagger + 延迟让属性卡片依次入场，与 shop/tasks/achievements 列表入场风格统一
      className="bg-cream border-2 border-ink px-4 py-3 flex items-center justify-between shadow-[3px_3px_0_#1a1a1a] card-hover animate-stagger"
      style={{ animationDelay: `${idx * 50}ms` }}
    >
```

**设计原因**：
- 与 `shop.tsx` 第 204 行 `${idx * 50}ms`、`tasks.tsx` 第 134 行 `${idx * 50}ms` 完全一致的延迟系数，全站列表入场节奏统一
- `UPGRADE_FIELDS` 是固定 6 个常量数组，不会动态变化，stagger 不会因数据刷新导致闪烁
- 50ms 延迟 × 6 项 = 250ms 总入场时长，落在"用户能感知但不拖沓"的体验区间
- 保留外层 `animate-fadeIn` 整组淡入，叠加内层 stagger 后形成"标题先现 → 卡片依次入场"的层次节奏

## 设计原则总结

本次优化遵循以下原则：

1. **状态语义化**：进度条的流光动画应反映状态语义——"进行中/可领取"才需要流光暗示活跃，"已领取/归档"应静态化。动画是信息，不是装饰
2. **辅助元素柔和化**：spinner 是辅助指示器，对比度应弱于主体内容，遵循"主体硬朗 + 辅助柔和"的 Neo-brutalism 层次
3. **交互一致性**：emoji hover 放大、列表 stagger 入场等微交互应在全站统一，避免"此页有反馈、彼页无反馈"的体验断层
4. **功能零侵入**：所有修改限于样式层（className、style、JSX 注释），业务代码一字不动；前几轮建立的调色板/响应式/无障碍基础完整保留
5. **小步增量**：本轮聚焦 4 类小改动（bug 修复 + 3 项一致性优化），每项独立可验证，遵循小步重构原则
