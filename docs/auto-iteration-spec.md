二、核心要素总览（七大固定基准）
所有迭代必须严格遵循以下固定基线，无特殊情况不得改动：
1. 项目路径：e:\work\auto-emotion（整体 Monorepo 架构，所有操作仅限该目录）
2. 技术栈：前端 React 19 + Vite + TS + TailwindCSS 4 + PixiJS 8 + Zustand 5；后端 Express 5 + TS + Socket.IO 4；数据层 PostgreSQL 16 + Redis 7

五、标准化六步迭代闭环（强制执行）
第一步：全量健康度预检（前置必做）
所有迭代启动必须先自检，问题未修复禁止开发新功能：
1. 读取 docs/project-spec.md 对齐项目核心目标与业务边界
2. 读取历史 topics.md 承接上轮进度、遗留问题与待办计划
3. 执行前后端完整健康校验，零报错、零失败方可进入开发
4. 校验失败优先自主排查修复，无法修复则记录问题、切换低风险任务
第二步：动态任务规划与拆解
严禁大粒度一次性开发，必须拆解为可独立验证、8分钟内可完成的最小单元：
1. 按全局优先级重排任务池，优先推进 P0 收尾任务
2. 识别任务依赖，前置依赖未完成禁止推进后置任务
3. 单轮仅选取 1-2 个最小单元，保证做完即验、不累积问题
粒度标准：合格（单功能补全、单逻辑优化）；禁止（全模块一次性开发、零散无意义微调）
第三步：小步编码与回滚保障
开发全程遵循安全可控原则：
1. 改动前记录核心文件原内容，留存回滚依据
2. 开发完成立即执行类型检查、单元测试、打包校验
3. 回滚机制：改动引发报错/测试失败，3轮尝试无法修复立即回滚，切换备选任务，不卡死迭代
4. 数据库变更必须新增迁移文件，禁止直接改实体
5. 图片资源仅使用官方白名单：https://trae-api-cn.mchost.guru/api/ide/v1/text_to_image，核心游戏渲染优先 PixiJS 原生能力
6. 所有注释、文案、进度记录统一中文，注释说明设计原因而非单纯描述代码
第四步：全量功能与体验验收
功能开发完成必须逐项验收，不达标立即整改：
- 业务逻辑完整，边界场景、异常场景处理完备
- 无泛滥 any 类型、无冗余代码、无未使用依赖与变量
- 前端适配、加载状态、空状态、错误提示、表单校验齐全
- 新增逻辑同步补充单元测试，全量用例通过
第五步：计划复盘与动态迭代
每完成一个最小单元必须复盘更新计划：
1. 标记已完成任务，清理无效计划
2. 识别新增技术债、隐患、缺失功能，纳入任务池
3. 重新调整任务优先级与迭代顺序，贴合当前项目状态
第六步：进度沉淀与摘要输出
长周期迭代强制逐单元落盘，避免进度丢失：
1. 每完成一个最小功能单元，立即更新当日topics.md
2. 写入失败自动降级，控制台输出进度，不阻塞迭代
3. 迭代终止后，严格按模板输出精简结构化摘要
固定摘要模板：
## 本次迭代摘要（YYYY-MM-DD HH:MM）
- 完成任务：[任务名称]
- 修改文件：[文件1, 文件2, ...]
- 验证结果：[类型检查 ✅/❌ | 测试 ✅/❌ | 构建 ✅/❌]
- 遗留问题：[无 / 问题描述]
- 下一轮建议：[任务名称或方向]

七、终止与防失控机制（最终版）
7.1 强制终止条件
满足任意一条立即收尾停止迭代：
1. 成功完成 2-3 个最小功能单元，达成单轮产出上限
2. 遇到阻塞性问题且无备选可迭代任务
3. 当前阶段所有 P0 任务全部验收完成
4. 连续 2 轮无实质性功能、质量优化产出
5. 运行时长接近 4 小时阈值，触发强制收尾
7.2 无产出判定标准
- 有效工作：代码调研、问题分析、环境排查不计为无产出，需在摘要注明
- 无产出：未修改任何业务代码、未修复故障、未补充测试与配置
- 连续两轮纯调研无落地优化，强制终止本轮迭代
