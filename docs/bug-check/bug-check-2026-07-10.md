#### C-02 async 中间件抛出错误存在未被捕获风险

- **文件**：
  - [server/src/middleware/auth.ts](file:///e:/work/auto-emotion/server/src/middleware/auth.ts) 第 22-42 行
  - [server/src/middleware/rate-limit.ts](file:///e:/work/auto-emotion/server/src/middleware/rate-limit.ts) 第 17-40 行
- **问题描述**：`authMiddleware` 和 `rateLimit` 都是 `async` 函数，内部使用 `throw new AppError(...)` 抛出错误。项目使用 Express 5（`express: ^5.2.1`），虽然 Express 5 支持 async 中间件错误自动传递给错误处理中间件，但该行为依赖错误处理中间件被正确注册（见 C-03：当前 `app.ts` 未注册 `errorHandler`，使用的是内联错误处理）。在当前配置下，async 中间件抛出的错误可能无法被正确映射为对应的 HTTP 状态码，且与 C-03 问题叠加会放大影响。
- **修复建议**：使用 `try/catch` 包裹并调用 `next(err)`：
  ```typescript
  export async function authMiddleware(req: Request, _res: Response, next: NextFunction): Promise<void> {
    try {
      // 验证逻辑
      next();
    } catch (err) {
      next(err);
    }
  }
  ```

#### C-03 app.ts 未使用 errorHandler 中间件，AppError 被降级为 500

- **文件**：
  - [server/src/app.ts](file:///e:/work/auto-emotion/server/src/app.ts) 第 144-158 行
  - [server/src/middleware/error-handler.ts](file:///e:/work/auto-emotion/server/src/middleware/error-handler.ts)（定义了完善的 errorHandler 但未被使用）
- **问题描述**：`app.ts` 中使用了一个内联的错误处理中间件，对所有错误都返回 `code: 500`。而 `error-handler.ts` 中定义了完善的 `errorHandler`（能按 `ErrorCode` 映射 HTTP 状态码），但完全未被注册使用。导致 `AppError` 的错误码语义丢失（401 变成 500，409 变成 500），前端无法基于 HTTP 状态码做差异化处理（如 401 跳登录），且 `error-handler.ts` 成为死代码。
- **修复建议**：删除 `app.ts` 中的内联错误处理，改为注册 `errorHandler`：
  ```typescript
  import { errorHandler } from './middleware/error-handler.js';
  app.use(errorHandler);
  ```

#### C-04 WebSocket room-manager 中房间操作存在 Redis 读-改-写竞态条件

- **文件**：[server/src/websocket/room-manager.ts](file:///e:/work/auto-emotion/server/src/websocket/room-manager.ts)
- **问题行号**：`joinRoom`（第 83-103 行）、`leaveRoom`（第 106-125 行）、`setReady`（第 128-144 行）、`setMode`（第 147-155 行）、`submitStress`（第 158-165 行）、`updateRoomStatus`（第 295-302 行）
- **问题描述**：这些方法都遵循 `getRoom` → 修改内存对象 → `setex` 写回的模式，这不是原子操作。当多个玩家同时操作同一房间时（如同时准备、同时加入），并发请求都会读到相同的旧数据，各自修改后写回，后写的会覆盖先写的修改，导致玩家被重复添加或丢失、准备状态丢失、房主转移错误。只有 `startGame`（第 178-180 行）使用了 `SET NX EX` 锁，其他方法都没有保护。
- **修复建议**：对所有房间写操作使用 Redis 的 `WATCH/MULTI/EXEC` 乐观锁，或使用 Lua 脚本保证原子性，或对每个操作使用 `SET NX EX` 分布式锁。

#### C-05 WebSocket handleDisconnect 不清理房间玩家，导致房间状态污染

- **文件**：[server/src/websocket/handlers.ts](file:///e:/work/auto-emotion/server/src/websocket/handlers.ts) 第 221-229 行
- **问题描述**：`handleDisconnect` 只广播 `PLAYER_OFFLINE` 通知，但**不从 `room.players` 中移除断线玩家**。导致房间内长期保留已离线玩家数据，`room.players.length` 不准确，其他玩家看到的房间状态包含离线玩家。如果房主断线，房主转移逻辑不会触发（因为 `leaveRoom` 没被调用），房间只能依靠 5 分钟 Redis TTL 自然过期。
- **修复建议**：在 `handleDisconnect` 中调用 `roomManager.leaveRoom` 清理玩家，或设计延迟清理机制（给重连窗口后自动移除）。

#### C-08 前端 battle.tsx/demo.tsx 未调用 BattleScene.destroy()，导致 PixiJS 纹理泄漏

- **文件**：
  - [client/src/pages/battle.tsx](file:///e:/work/auto-emotion/client/src/pages/battle.tsx) 第 309-334 行

#### C-09 前端 battle.tsx cleanup 中 leaveRoom 未做异常保护，可能阻断资源释放

- **文件**：[client/src/pages/battle.tsx](file:///e:/work/auto-emotion/client/src/pages/battle.tsx) 第 326 行

#### L-02 前端 boss-game/brawl-game 可破坏物纹理未缓存

- **文件**：
  - [client/src/game/games/boss-game.ts](file:///e:/work/auto-emotion/client/src/game/games/boss-game.ts) 第 185-200 行
  - [client/src/game/games/brawl-game.ts](file:///e:/work/auto-emotion/client/src/game/games/brawl-game.ts) 第 153-157 行
- **问题描述**：创建可破坏物时每次都调用 `this.app.renderer.generateTexture()` 生成新纹理，不像投射物和玩家纹理那样懒加载复用。多次重开游戏会反复生成相同纹理，造成 GC 抖动。
- **修复建议**：将可破坏物纹理按颜色缓存，或使用 `AssetLoader` 的 `createRectTexture` 方法。

#### L-03 前端 boss-game/brawl-game 的 init 声明为 async 但无异步操作

- **文件**：
  - [client/src/game/games/boss-game.ts](file:///e:/work/auto-emotion/client/src/game/games/boss-game.ts) 第 150 行
  - [client/src/game/games/brawl-game.ts](file:///e:/work/auto-emotion/client/src/game/games/brawl-game.ts) 第 149 行
- **问题描述**：`init` 方法声明为 `async` 但内部全是同步操作。调用方 `battle-scene.ts` 也没有 `await`。如果未来在 `init` 中添加异步资源加载，调用方不 await 会导致竞态条件。
- **修复建议**：移除 `async` 声明，或调用方添加 `await`。

#### L-04 前端 speed-game.ts update 每帧调用 updateHUD()

- **文件**：[client/src/game/games/speed-game.ts](file:///e:/work/auto-emotion/client/src/game/games/speed-game.ts) 第 586 行
- **问题描述**：`update` 方法每帧都调用 `this.updateHUD()`，即使分数、连击、时间都没变化。PixiJS Text 的 `text` 属性赋值会触发重新渲染文本纹理，每帧设置会造成不必要的性能开销。
- **修复建议**：只在数据变化时更新 HUD，或对比新旧值后赋值。

#### L-05 前端 ErrorBoundary.tsx 用 new Error(componentStack) 包装组件栈

- **文件**：[client/src/components/ErrorBoundary.tsx](file:///e:/work/auto-emotion/client/src/components/ErrorBoundary.tsx) 第 26 行
- **问题描述**：用 `new Error()` 包装字符串只是为了满足 `logger.error` 的 `unknown` 参数类型，语义上不太合适，stack trace 会显示为 `Error: <componentStack>` 而非真正的调用栈。
- **修复建议**：直接将组件栈作为字符串拼接：`logger.error('Component stack: ' + errorInfo.componentStack)`。

#### L-06 前端 PressureRadar.tsx 两个重叠的 circle 绑定相同事件

- **文件**：[client/src/components/PressureRadar.tsx](file:///e:/work/auto-emotion/client/src/components/PressureRadar.tsx) 第 175-196 行
- **问题描述**：主圆和透明扩大点击区域圆都绑定了 `onMouseEnter`/`onMouseLeave`，可能在某些边界情况下触发意外的事件序列，导致 tooltip 闪烁。
- **修复建议**：只在透明圆上绑定事件，主圆仅作为视觉展示。

#### L-07 后端 ai/client.ts 没有重试机制

- **文件**：[server/src/ai/client.ts](file:///e:/work/auto-emotion/server/src/ai/client.ts) 第 21-50 行
- **问题描述**：AI 服务调用失败直接抛错，没有重试。虽然 `room-manager` 有兜底数据，但对于 `POST /api/ai/monster` 这种直接返回给客户端的接口，一次失败就返回 500。
- **修复建议**：对网络超时类错误重试 1-2 次，使用指数退避。

#### L-08 后端 utils/idempotency.ts 在 Redis 故障时直接抛错

- **文件**：[server/src/utils/idempotency.ts](file:///e:/work/auto-emotion/server/src/utils/idempotency.ts) 第 7-15 行
- **问题描述**：`checkIdempotency` 在 Redis 不可用时抛错，导致请求失败。但 `user-service.refreshToken` 中对 Redis 黑名单检查采用了降级策略，幂等性检查应考虑类似的降级策略。
- **修复建议**：提供 `checkIdempotencySafe` 变体，Redis 故障时返回 true（放行）并记录警告日志。

#### L-09 后端 app.ts 缺少优雅关闭处理

- **文件**：[server/src/app.ts](file:///e:/work/auto-emotion/server/src/app.ts)
- **问题描述**：没有注册 `process.on('SIGTERM')` / `process.on('SIGINT')` 处理逻辑。进程被杀死时 PostgreSQL 连接池不会优雅关闭，Redis 连接不会断开，HTTP 服务器不会停止接收新连接。
- **修复建议**：添加优雅关闭逻辑：
  ```typescript
  process.on('SIGTERM', async () => {
    httpServer.close();
    await pool.end();
    redis.disconnect();
    process.exit(0);
  });
  ```

#### L-10 后端 logger.ts 没有日志级别控制

- **文件**：[server/src/utils/logger.ts](file:///e:/work/auto-emotion/server/src/utils/logger.ts)
- **问题描述**：所有 `debug/info/warn/error` 日志都直接输出到 console，没有基于环境变量的级别过滤。生产环境下 `debug` 日志会产生大量输出。
- **修复建议**：根据 `config.nodeEnv` 或 `LOG_LEVEL` 环境变量过滤低于设定级别的日志。

#### L-11 后端 WebSocket 连接时未检查 JWT 黑名单

- **文件**：[server/src/websocket/index.ts](file:///e:/work/auto-emotion/server/src/websocket/index.ts) 第 36-47 行
- **问题描述**：WebSocket 握手鉴权只验证 JWT 签名，不检查 Redis 黑名单。用户登出后（logout 函数会将 token 加入黑名单），仍可用旧 token 建立 WebSocket 连接。HTTP 路由的 `authMiddleware` 会检查黑名单，但 WebSocket 不会。
- **修复建议**：在 `io.use` 中间件中增加 Redis 黑名单检查。

## 六、检查结论

本次检查共发现 **52 个问题**，其中 **9 个严重**、**13 个高**、**19 个中**、**11 个低**。

虽然所有自动化测试（前端 223 + 后端 614 = 837 个用例）和构建均通过，但静态代码审查发现了大量潜在 bug，主要集中在以下领域：

1. **认证与错误处理**（后端）：路由未挂载 authMiddleware、errorHandler 未注册，导致生产环境接口不可用且错误码语义丢失。测试通过是因为测试环境使用了模拟中间件，与生产环境不一致，存在**测试与生产环境差异风险**。

2. **游戏逻辑**（前端）：speed-game 的撕胶带功能完全失效、西瓜粒子内存泄漏、BattleScene 纹理未销毁、得分归属错误等问题，直接影响游戏体验和资源管理。

3. **并发安全**（后端）：WebSocket 房间管理存在读-改-写竞态、idle 结算无幂等保护、好友/宠物服务事务不完整，可能导致数据不一致或资产刷取。

4. **资源管理**（前端）：PixiJS 纹理、ParticleEffect、React Root 等资源未正确销毁，长期运行会导致内存/GPU 内存泄漏。

**建议在修复 C-01 ~ C-09 这 9 个严重问题之前，暂停新功能开发与部署。**
